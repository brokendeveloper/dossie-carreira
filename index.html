Projeto Integrador — 5º Semestre

LicitaME

5º Semestre · 2026.1 · Python ETL, MongoDB Atlas, FastAPI, Expo

Em andamento
Plataforma mobile que conecta Micro Empreendedores Individuais (MEIs) a oportunidades de licitação pública. Backend em FastAPI com dados governamentais do PNCP (Portal Nacional de Contratações Públicas) processados via ETL Python e armazenados em MongoDB Atlas. Frontend mobile em Expo (React Native). Público-alvo: MEIs que não têm estrutura para monitorar licitações manualmente.
🎭
Meu papel na equipe
Membro da equipe de ETL/data engineering. Responsável pelo pipeline de ingestão de dados do PNCP: extração via API pública, transformação e normalização dos dados (formatos heterogêneos por órgão), carregamento no MongoDB Atlas para consumo pela API FastAPI. Também participei do alinhamento do schema do MongoDB com o time de API para garantir que os dados estivessem estruturados para leitura eficiente, evitando retrabalho. Colaborei com o time de frontend para entender as necessidades de dados das telas e ajustar o ETL conforme necessário. Participei também da criação da API de consulta, garantindo que os endpoints atendessem aos requisitos do frontend e fossem eficientes para o volume de dados esperado.
📦
Minhas entregas concretas
Pipeline ETL em Python para extração e transformação de dados do PNCP. Modelagem dos documentos no MongoDB Atlas para leitura eficiente pela API FastAPI. Mapeamento das inconsistências de formato entre órgãos governamentais diferentes. Integração entre o ETL e o schema da API, alinhamento feito em reunião documentada para evitar retrabalho. Construção de API de consulta em FastAPI para servir os dados processados. Documentação técnica do pipeline e da API para facilitar manutenção futura. Colaboração com o time de frontend para garantir que os dados entregues pela API atendessem às necessidades das telas, ajustando o ETL conforme necessário.
🔥
Dificuldades e como lidei
Volume de dados do PNCP inconsistente: cada órgão usa formatos de resposta diferentes, o que quebrava o pipeline silenciosamente. Resolvi com validação de schema no ETL antes de carregar. Tensão entre o time de ETL e o time de API sobre o schema do MongoDB: propus uma reunião de alinhamento antes da integração e documentamos a decisão para evitar retrabalho.

Carta Profissional

Querido Luccas, Você entrou no 5º semestre como estagiário numa empresa que trabalha com um sistema legado em Delphi, o tipo de stack que a maioria dos colegas de curso nunca vai tocar. E você foi. Sem reclamar, sem romantizar, sem fingir que era o trabalho dos sonhos. Você foi porque sabia que sistema legado de alta criticidade ensina coisas que nenhum tutorial ensina: como ler código sem documentação, como medir o impacto de uma mudança antes de subir, como conviver com a incerteza de não saber exatamente o que vai quebrar. Você saiu desse semestre como Associate. Não pediu. Foi promovido. O TechLead escreveu que você é "fácil de lidar, aprendizado rápido, cada vez mais importante para o time". Guarde esse texto. Não porque valida você, mas porque é evidência de como outros te veem quando você está sendo você mesmo, sem performance. Neste mesmo semestre você trabalhou no LicitaME: começou pelo ETL, foi puxado para a API, depois para as telas mobile. Você poderia ter ficado no seu escopo. Não ficou. Isso diz algo sobre como você trabalha em equipe: você se move para onde a necessidade está, não apenas para onde foi designado. O dossiê te forçou a nomear o que você nunca tinha nomeado com clareza: cloud ausente do portfólio, microsserviços teórico, inglês oral travando em tempo real, CS fundamentals como ponto cego em entrevistas técnicas. Saber o nome do problema é metade da solução. Você saiu do semestre sabendo os nomes. O que ainda não está resolvido: inglês oral. Cloud. Microsserviços em produção real. Não finjas que estão, eles não estão. Mas agora você tem um plano com prazo, não só uma intenção. Uma coisa que vale carregar: você construiu evidências reais enquanto ainda estava na faculdade. Não projetos de brinquedo. Sistemas em produção, com PL/SQL de alta criticidade, com Clean Architecture em código que outras pessoas dependem. Isso é raro. Não pela raridade em si, mas pelo que ela revela: que você não espera as condições ideais para começar. Continue assim. Mas fecha o gap de cloud logo. Luccas, Jun/2026
